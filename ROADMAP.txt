QubitOS Development Roadmap
===========================


Overview
--------

QubitOS is an open-source quantum control kernel providing Hamiltonian-level
pulse optimization, hardware abstraction, and calibration management for
quantum computing research. The monorepo contains three modules: proto/
(Protocol Buffer definitions), hal/ (Rust HAL server), and core/ (Python
client, optimizer, and calibration). v0.5.0 is complete; all five
architecture gaps addressed, Rust-native GRAPE and Lindblad solver
operational, IBM/AWS/IQM backends integrated. Next milestone: v0.6.0.


CI/CD-first development policy
------------------------------

All development follows a CI/CD-first approach. This is not negotiable.

The CI/CD contract:

1. Green CI is a prerequisite for any phase transition.
   No phase can begin until the previous phase's CI is passing.
   No commits to main if CI is red (except CI fixes).

2. Local validation before push.
   Run scripts/ci-check.sh before every push.
   Pre-commit hooks must be installed and active.

3. No bypassing branch protection.
   Branch protection rules exist for a reason.
   If you must bypass, you MUST fix CI in the next commit.

CI check commands:

    # hal (Rust)
    cd hal
    cargo fmt --check && cargo clippy -- -D warnings && cargo test

    # core (Python)
    cd core
    source .venv/bin/activate
    ruff check src/ && ruff format --check src/ && pytest tests/

    # proto
    cd proto
    buf lint && buf format -d --exit-code
    python -m build

time model (GAP 1), no cumulative error tracking (GAP 2), Python-only solver

leaking into the Hamiltonian-first API (GAP 5). Multi-qubit expansion
(v0.3.0) would have amplified each of these into serious problems. v0.2.0
hardens the foundations before building the next floor.

v0.6.0: Stochastic Master Equation Solver
-----------------------------------------

Theme: from ensemble averages to single quantum trajectories.
Thesis connection: mathematical foundation for measurement-based feedback
control (MS thesis core).

The Lindblad equation (v0.5.0) computes ensemble-averaged dynamics. The SME
conditions state evolution on a continuous measurement record, producing a
conditional density matrix rho_c(t); our best knowledge of the quantum
state given everything we have measured.

Design spec: core/docs/specs/SME-FEEDBACK-SPEC.txt

Release-gating path: the v0.6.0 release ships on the Rust solver with
Rayon-parallel ensemble. The bare-metal C fast path (0.6.4) is a stretch
track tied to the LANL QCSS summer 2026 deliverable; it is bonus
performance for v0.8.0's compute budget, not a dependency for cutting
v0.6.0.

Validation backbone: every numerical surface introduced in this version is
decorated through the project's correctness checker before it lands.
Density matrices emitted per timestep are validated for Hermiticity,
positive-semidefiniteness, and trace preservation (with stochastic-
tolerance override for SME trajectories). See the "Validation Decoration"
deliverable in each sub-phase.


0.6.1 SME solver (Python + Rust)

  [ ] Euler-Maruyama integrator for Ito SDE (Wiseman & Milburn 2009, Ch. 4)
  [ ] Measurement superoperator H[c]rho = c rho + rho c-dagger
      - Tr[(c+c-dagger)rho] rho
  [ ] Homodyne measurement record simulation:
      I(t) dt = sqrt(eta) Tr[(c+c-dagger) rho_c] dt + dW
  [ ] Configurable measurement efficiency eta in [0,1]; eta=0 reduces to
      Lindblad, eta=1 is perfect detection
  [ ] Single trajectory solver with optional full state history
  [ ] Monte Carlo ensemble solver with statistics (mean, variance,
      convergence diagnostics)
  [ ] Adaptive timestep (trace-norm monitoring)
  [ ] Positivity monitoring and optional projection onto positive cone
  [ ] Proto messages: SMEConfig, SMEResult
  [ ] Validation decoration: per-timestep density matrix output validated
      for Hermiticity, PSD, and trace preservation at the module boundary
      (stochastic-tolerance override applied to single-trajectory output;
      strict tolerance applied to ensemble-averaged output)


0.6.2 Validation suite

  [ ] eta=0 reduces exactly to Lindblad (numerical identity, atol=1e-10)
  [ ] Ensemble average (N to infinity) converges to Lindblad at rate
      1/sqrt(N)
  [ ] Analytical benchmark: driven qubit steady-state purity (Wiseman 1994)
  [ ] Analytical benchmark: spontaneous emission quantum jump signature
      (eta=1)
  [ ] Cross-validation against QuTiP smesolve (ensemble expectations,
      atol=0.05)
  [ ] Golden file tests for reproducibility (seed-deterministic
      trajectories)


0.6.3 Rust SME solver

  [ ] src/sme/ module: measurement.rs, integrate.rs, trajectory.rs
  [ ] PyO3 bindings: RustSMESolver Python class (drop-in for Python solver)
  [ ] Thread-parallel ensemble via Rayon
  [ ] Benchmark: >= 5x speedup over Python SME


0.6.4 Bare-metal Lindblad fast path (stretch; LANL summer 2026)

Status: stretch track, not release-gating for v0.6.0. The Rust solver with
Rayon-parallel ensemble is the v0.6.0 release path. This subsection lands
when the LANL summer deliverable lands; if it slips past v0.6.0, it ships
in a later patch release or rolls into v0.8.0's compute campaign without
blocking v0.7.0.

Context: the LANL QCSS summer project produces a standalone C library
(lindblad/) for SIMD-optimized Lindblad propagation. This subsection
integrates it into QubitOS as a high-performance backend for the existing
solver stack.

Why: the systematic v0.8.0 experiments (5 gates x 50 noise points x 4
methods x 1000 trajectories) benefit from the C fast path's speedup over
Rust+Rayon. Inside v0.6.0 itself, the Rust solver is sufficient for the
validation suite and the single-trajectory / ensemble-of-thousands API
surface.

Architecture (two-tier solver dispatch):

    QubitOS Python API
        |
        v
    solver.evolve(H, rho0, tlist, c_ops)
        |
        v
    [dispatch logic]
        |
        +-- d <= 27 AND time-independent  --> Tier 2: C fast path (FFI)
        |     - Precompute propagator P = exp(L * dt) via Pade [13/13]
        |     - SIMD matvec per timestep (AVX2/AVX-512)
        |     - OpenMP-parallel batch sweeps
        |     - Returns NumPy array (zero-copy)
        |
        +-- otherwise                       --> Tier 1: general Rust solver
              - Adaptive RK4/RK45 integration
              - Sparse matrix support
              - Arbitrary dimension

Deliverables (stretch):

  [ ] FFI bridge: extern "C" bindings in hal/src/lindblad/ffi.rs
  [ ] C library build integration: CMake subproject or pkg-config discovery
  [ ] Automatic dispatch: dimension check + time-independence check in
      Python solver
  [ ] Validation: C fast path matches Rust solver to machine precision
      (atol=1e-12)
  [ ] Benchmark: >= 50x speedup for d=3 robustness landscapes vs current
      Rust solver
  [ ] Batch sweep API: solver.batch_sweep(H_func, param_grid, rho0,
      target) returning 2D fidelity array
  [ ] Fallback: graceful fallback to Rust solver with warning if C library
      not installed
  [ ] Proto extension: SolverBackend enum field on LindbladConfig (AUTO,
      RUST, C_FAST_PATH)

Future (PhD scope, not v0.6.0):

    Tier 3: FPGA fast path
        - Same propagator algorithm, fixed-point arithmetic
        - Deterministic ~35-70 ns per timestep
          (vs ~50-100 ns C, vs ~10-30 ms QuTiP)
        - USB/SPI host interface for propagator upload
        - Real-time feedback capable (sub-microsecond latency)

v0.7.0: Lyapunov Feedback Controller
------------------------------------

Theme: close the loop.
Thesis connection: the core technical contribution; the first open-source
continuous quantum feedback controller with production-grade engineering.


0.7.1 Controller design

  [ ] Lyapunov function: V(rho_c) = 1 - Tr[rho_target * rho_c]
  [ ] Single-axis feedback law:
      delta-Omega(t) = -K * Tr[rho_target * [i*sigma_k/2, rho_c]]
  [ ] Multi-axis (full SU(2)) feedback: independent gains K_x, K_y, K_z on
      three generators
  [ ] Configurable feedback delay (tau_fb >= 0; realistic values
      ~10-100 ns)
  [ ] Amplitude saturation: |delta-Omega(t)| <= delta-Omega_max (hardware
      DAC constraint)
  [ ] Feedback energy tracking: integral of |delta-Omega(t)|^2 dt
  [ ] Proto messages: FeedbackConfig, FeedbackResult
  [ ] Validation decoration: Lyapunov function trajectory V(t) validated
      for finiteness and bounded range; ensemble mean V(t) checked for
      monotone-in-mean property at the module boundary


0.7.2 Integration with SME

  [ ] solver.solve_with_feedback(); SME step then controller step then
      update Hamiltonian
  [ ] Feedback-controlled trajectory with full correction history
  [ ] Feedback-controlled ensemble for statistical analysis
  [ ] Provenance: feedback parameters recorded in Merkle tree


0.7.3 Comparison framework

  [ ] noise_sweep_comparison(); fidelity vs noise strength for GRAPE, DRAG,
      Gaussian, Lyapunov
  [ ] Crossover point detection: gamma* where feedback fidelity =
      GRAPE fidelity
  [ ] Publication-quality plotting (matplotlib, reproducible scripts)
  [ ] Tabular output for paper inclusion


0.7.4 Stability analysis tools

  [ ] Lyapunov function trajectory visualization (V(t) for individual and
      ensemble-averaged trajectories)
  [ ] Bloch sphere trajectory visualization (rho_c(t) on the Bloch sphere
      with feedback arrows)
  [ ] Convergence rate estimation (exponential fit to V(t))


v0.7.0 exit criteria

  * Feedback controller; Lyapunov, single + multi-axis, configurable
    gain/delay.
  * Zero gain equals open loop; numerical identity with zero-gain feedback

  * V-dot < 0 on average; Lyapunov function non-increasing in ensemble

  * Crossover plot; >= 1 comparison figure (GRAPE vs feedback vs noise).

Risk mitigation
---------------

CI drift                    Mandatory local checks, pre-commit hooks,
                            golden rule enforcement.
QuTiP version incompatibility   Pin version, test on updates, long-term
                                migrate to bare-metal C + Rust solvers.
IQM API changes             Abstract behind backend trait, version client,
                            integration tests.
Numerical instability in GRAPE   Extensive testing, gradient clipping,
                                 golden file regression tests.
Performance bottlenecks (multi-qubit)   Profile early, sparse matrices,
                                        benchmark at each qubit count.
Scope creep                 Strict phase boundaries, defer features,
                            design specs before code.
Time model complexity       Start with minimal constraints, extend
                            incrementally.
Rust ecosystem gaps         Evaluate libraries early, prototype matrix
                            exp before committing.
Backward compatibility breaks   Deprecation aliases for one release
                                cycle, semver discipline, migration guides.
SME numerical stiffness     Adaptive timestep, trace-norm monitoring,
                            Milstein fallback, QuTiP validation.
Stochastic convergence      Report confidence intervals, require >= 1000
                            trajectories for published results.
Lyapunov stability proof gaps   Work with Drakunov on LaSalle invariance;
                                cite Mirrahimi & van Handel 2007.
3-level Hilbert space scaling   d=3 is small; only a concern at d >= 64
                                (already handled by sparse expm).
Thesis timeline pressure    v0.6.0 is self-contained; v0.7.0 builds on
                            it; each version is independently publishable.

Development principles
----------------------

Code quality:

  * All code reviewed before merge.
  * CI must pass (all enabled jobs green).
  * Test coverage targets enforced per phase.
  * Documentation required for all new features.
  * Functions <= 50 lines (Python), <= 60 lines (C++/Rust), single
    responsibility.

Reproducibility:

  * Every result traceable to code + seed + calibration via provenance
    hash.
  * Golden file tests for all numerical code.
  * Pinned dependencies in lock files.
  * Provenance Merkle tree covering calibration, pulse sequence,
    optimizer config, software versions.

Physics correctness:

  * Cite papers for all non-trivial physics (DOI or arXiv ID in code
    comments and docs).
  * Validate unitarity of all gate/unitary constructions.
  * Validate normalization of all quantum states.
  * Validate physical bounds (amplitudes, frequencies, durations).
  * Decoherence budget tracked and reported; never silently exceeded.
  * Error accumulation modeled across pulse sequences, not just
    per-gate.


Long-term vision: bare-metal solver stack
-----------------------------------------

Added March 2026.

QubitOS's solver stack is evolving toward eliminating Python numerical
dependencies (QuTiP, SciPy) from the performance-critical path. The
trajectory:

    v0.1.0 to v0.4.0    QuTiP via PyO3      Python    Retired as primary
    v0.5.0              Rust ndarray solver Rust      Current production
    v0.6.0              C bare-metal fast path  C     In progress; LANL Summer 2026
    Post-thesis         FPGA fast path      Verilog   Planned; PhD scope

Why this matters:

  * QuTiP remains as a validation oracle (testing only), never in the
    hot path.
  * Each tier is independently validated against the tier above it.
  * The C fast path is a standalone open-source artifact (lindblad/
    library), usable outside QubitOS.
  * The FPGA tier uses the same fixed-size propagator algorithm as the C
    tier.
  * The architecture supports future GPU acceleration (CUDA/ROCm) as an
    additional backend.

End state: QubitOS's simulation engine is a multi-backend dispatcher that
selects the fastest available solver for the problem size and available
hardware, with all backends validated to machine precision against a
common reference.
